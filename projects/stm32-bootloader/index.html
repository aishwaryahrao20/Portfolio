STM32 Bootloader & Firmware Update System

A fault-tolerant bare-metal bootloader implemented in C targeting ARM Cortex-M4, featuring a 6-state boot state machine, CRC32 image verification, sector-level flash programming, and a chunked UART firmware update protocol — designed from scratch without OS or HAL abstractions.

Overview

This project implements a production-pattern STM32 bootloader from scratch in bare-metal C with no OS, no vendor HAL abstractions, and no external libraries — only the processor, flash, UART, and a precisely defined flash memory map. The bootloader manages the entire boot sequence: detecting a firmware update trigger, receiving a chunked binary over UART, verifying per-packet CRC16 integrity and full-image CRC32 correctness, programming the verified image into flash, and branching to the application's reset handler via a direct stack-pointer and program-counter jump.

The simulation backend uses a 512 KB file-backed flash model and file-descriptor UART, enabling full-cycle testing on macOS/Linux before hardware deployment on a NUCLEO-F446RE.

Boot State Machine

The bootloader implements a six-state deterministic state machine with a fault-safe fallback on image corruption.

  • INIT: Prints flash memory layout and initializes all peripherals.
  • CHECK_TRIGGER: Reads a GPIO pin or update-trigger file — routes to UPDATE_MODE or VERIFY_APP.
  • UPDATE_MODE: Receives firmware binary over UART in 128-byte chunks, ACKs each packet on CRC16 pass, NACKs and retransmits on failure, then flashes the staging area.
  • VERIFY_APP: Computes a full CRC32 over the staged application image and cross-checks against stored metadata (magic, version, size, CRC32).
  • JUMP_TO_APP: Loads the application's initial stack pointer from flash offset 0, relocates the vector table, and branches to the reset handler via inline assembly.
  • ERROR: CRC32 mismatch — stays in bootloader indefinitely and awaits a new firmware image, preventing corrupt firmware from executing.

Flash Memory Layout

The 512 KB flash space is partitioned into four fixed regions to enable independent bootloader write-protection, metadata management, dual-bank staging, and safe fallback.

  • 0x08000000 – Bootloader (32 KB): Write-protected; contains the boot state machine, CRC logic, and UART receiver.
  • 0x08008000 – App Metadata (4 KB): Stores magic number, firmware version, image size, and expected CRC32 — verified on every boot.
  • 0x08009000 – Application (128 KB): Active motor control firmware partition, branched to after successful verification.
  • 0x08029000 – Staging Area (128 KB): Receives incoming firmware during an update cycle; copied to the application partition only after full CRC32 verification.

UART Firmware Update Protocol

The binary update protocol transmits firmware as 128-byte data chunks wrapped in an 8-byte header carrying sequence number, length, and per-packet CRC16. The bootloader ACKs each correctly received packet and NACKs any packet with a CRC16 mismatch, triggering host-side retransmission. After all packets are received, the bootloader copies the staged image to the application partition, writes verified metadata, and runs a final CRC32 check before the next boot.

The host-side Python tooling (host_updater.py) packages a binary into correctly framed packets with pre-computed CRC16 per chunk and sends them over the simulated UART.

CRC32 Implementation

CRC32 is implemented from scratch using polynomial 0xEDB88320 — the same polynomial used by the STM32 hardware CRC unit and the Ethernet standard — ensuring that the software implementation produces identical results to the hardware accelerator when deployed on-chip. CRC16 is applied per-packet for link-layer integrity; CRC32 is applied over the entire application image for boot-time verification.

Measured outcomes

  • Achieved 100% corrupt-image detection before boot by implementing full-image CRC32 verification (polynomial 0xEDB88320) in the VERIFY_APP state, cross-checked against metadata stored in a dedicated 4 KB flash sector — validated across blank-flash, valid-firmware, and deliberately corrupted image test cases.
  • Successfully received and flashed a 512-byte firmware image (4 packets × 128 bytes) with all packets ACK'd and a final verified CRC32 of 0x4FACB929, as measured by end-to-end simulation using host_updater.py and the file-backed flash model.
  • Demonstrated safe boot recovery by designing the ERROR state to hold the bootloader indefinitely on any CRC mismatch, preventing execution of a corrupt application image — tested by deliberately injecting a single-byte corruption into the staged binary.
  • Implemented a direct bare-metal application jump in C via inline assembly — setting the MSP from flash offset 0x00 and branching to the reset handler at offset 0x04 — without OS or HAL involvement, validated by a successful jump to 0x08009000 after normal-boot verification.
  • Enabled full cross-platform firmware update cycle testing on macOS/Linux by replacing UART and flash hardware with a file-descriptor UART and a 512 KB file-backed flash model, achieving functional parity with the STM32 hardware path for all four test scenarios.

Hardware porting path (STM32F446RE)

Porting to the NUCLEO-F446RE requires replacing four simulation primitives with their STM32 HAL equivalents: flash_sim_write() → HAL_FLASH_Program(), flash_sim_erase() → HAL_FLASHEx_Erase(), file-based UART → HAL_UART_Receive_IT(), and trigger file check → HAL_GPIO_ReadPin(). The boot state machine, CRC32, protocol parser, and jump logic are entirely unchanged.
