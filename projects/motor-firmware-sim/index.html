Motor Control Firmware Simulator with HAL-Portable Architecture

Closed-loop PID motor control firmware written in C, architected with a Hardware Abstraction Layer that enables identical application logic to run on macOS/Linux simulation and bare-metal STM32 without any changes above the HAL boundary.

Overview

This project implements a portable, closed-loop motor control firmware stack in C. The architecture is structured around a strict Hardware Abstraction Layer (HAL), enabling the application layer — PID controller, motor model, RTOS scheduler, and protocol handlers — to run identically on both a macOS/Linux simulation environment and bare-metal STM32 (ARM Cortex-M4) hardware. The simulation backend uses POSIX threads and printf, while the hardware backend maps directly to STM32 HAL calls with zero application-layer changes.

Architecture

The firmware is layered across three tiers: the Application Layer, the HAL Interface, and the platform-specific backends. The HAL interface exposes five protocol entry points — hal_uart_send, hal_i2c_read, hal_spi_write, hal_pwm_set_duty, and hal_encoder_get_ticks — allowing the application layer to remain entirely hardware-agnostic.

Porting to STM32F446RE requires replacing only hal_sim.c with hal_stm32.c and swapping POSIX thread tasks for FreeRTOS task definitions. Every PID, motor model, and protocol module remains unchanged.

RTOS Task Structure

The firmware runs four cooperative real-time tasks, each with a fixed period and a dedicated responsibility. Shared state between tasks is protected by a POSIX mutex, mirroring the FreeRTOS xSemaphoreTake pattern for direct hardware porting.

  • Encoder task (1 ms): Reads motor speed via HAL and updates the first-order motor model at 1 kHz.
  • PID task (5 ms): Computes closed-loop PID output every 5 ms and drives the PWM duty cycle via hal_pwm_set_duty.
  • CAN task (10 ms): Parses simulated CAN frames to update the speed setpoint at 100 Hz.
  • Telemetry task (50 ms): Streams UART telemetry logs and polls the I2C thermal sensor (TMP102 @ 0x48) and SPI motor driver (DRV8323) at 20 Hz.

Multi-Protocol Firmware Stack

The firmware integrates five hardware protocols across the system, each with a defined role and a clean simulation pathway.

  • UART: Real-time telemetry stream — simulated via printf to terminal, maps to HAL_UART_Transmit() on STM32.
  • SPI: Motor driver configuration (DRV8323) — logged as a transaction in simulation, maps to HAL_SPI_TransmitReceive().
  • I2C: Thermal monitoring (TMP102 @ 0x48) — driven by the software motor thermal model, maps to HAL_I2C_Mem_Read().
  • CAN: Speed setpoint delivery — parsed from simulated CAN frames with a binary protocol parser.
  • PWM: Motor drive signal — duty cycle fed directly into the motor model, maps to __HAL_TIM_SET_COMPARE().

PID Controller Design

The closed-loop controller is implemented with anti-windup clamping to prevent integrator saturation during transients. Tuned parameters: Kp = 0.8, Ki = 0.3, Kd = 0.02. Output range is 0–100% PWM duty cycle with the integral term clamped to the same bounds to bound output during large error excursions.

Measured outcomes

  • Achieved 1.6% steady-state RPM error (1,476 RPM vs. 1,500 RPM setpoint) by tuning closed-loop PID (Kp=0.8, Ki=0.3, Kd=0.02) with integral anti-windup clamping, as validated through Python step-response log analysis.
  • Achieved sub-500 ms rise time (0→90% setpoint) by scheduling the PID task at a fixed 5 ms period under a mutex-protected 4-task RTOS scheduler, confirmed via UART telemetry capture.
  • Maintained thermal steady-state of ~41°C at 50% load by integrating a first-order motor thermal model polled via I2C at 20 Hz, with real-time temperature streaming over UART.
  • Achieved zero application-layer code changes between simulation and hardware targets by enforcing a strict 5-function HAL boundary (UART, SPI, I2C, PWM, encoder), validated by mapping each HAL call 1:1 to its STM32 HAL equivalent.
  • Demonstrated a complete 5-protocol embedded stack (UART, SPI, I2C, CAN, PWM) running concurrently across 4 RTOS tasks with shared-state integrity maintained by POSIX mutex across 1 ms, 5 ms, 10 ms, and 50 ms task periods.

Hardware porting path (STM32F446RE)

The firmware is designed for direct deployment on NUCLEO-F446RE + DC gearmotor + L298N + TMP102 + SN65HVD230. Porting requires replacing the simulation backend with STM32 HAL calls for each of the five HAL functions and substituting POSIX thread tasks with FreeRTOS task definitions — all application, PID, and protocol logic is unchanged above the HAL boundary.
