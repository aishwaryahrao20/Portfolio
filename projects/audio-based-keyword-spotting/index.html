Embedded TinyML Systems on Arduino Nano 33 BLE Sense

Two end-to-end TinyML deployments on the same microcontroller — one that listens for spoken keywords, one that reads body motion — sharing a common embedded pipeline: sensor input → feature extraction → on-device neural inference → real-time output.

Platform & Shared Architecture

Both projects run on the Arduino Nano 33 BLE Sense — a thumb-sized microcontroller with a 64 MHz Cortex-M4 core, 256 KB RAM, and an onboard sensor suite that includes a PDM microphone, a 6-axis IMU (BMI270), and a 3-axis magnetometer (BMM150). No cloud, no host compute — every inference decision is made on the device itself in milliseconds.

The pipeline architecture is identical across both projects: collect sensor data → extract compact numerical features → normalize → run a small neural network → emit a discrete prediction over serial. What changes is the sensor, the feature space, and the model topology.

Project 1 of 2 · Audio

On-Device Keyword Spotting via Log-Energy MLP

What It Does

A real-time speech command recognizer that detects five absolutist keywords — "all," "must," "none," "never," and "only" — from a continuous microphone stream on the Arduino, with no host processor or network required. The entire recognition pipeline, from raw PCM samples to a predicted class label, runs in under 50 ms.

Dataset & Collection

A custom dataset of 250+ labeled 1-second WAV clips was collected across five recording experiments, capturing different speakers, distances, and acoustic environments. Clips were organized into keyword-named directories so the training pipeline could infer labels from folder names automatically. An OGG-to-WAV utility (convert_ogg_wav.py) standardized all audio to 16 kHz mono before any processing began.

The full dataset was randomly shuffled and split 80 / 10 / 10 into train, validation, and test subsets with stratification to maintain per-class balance. Oversampling on the training split corrected any class imbalance identified during the initial data audit.

Feature Extraction — Log-Energy Frames

Instead of full MFCC computation, a lightweight log-energy feature was designed to mirror what the Arduino's Cortex-M4 can compute between audio interrupts. Each clip is segmented into 101 frames; per frame, the mean absolute amplitude is log-transformed:

Et = mean(|xk|)  ⟶  Xt = log(1 + Et)

This scalar is replicated across 13 feature dimensions, producing a (101 × 13) tensor — exactly matching the kFeatureTimeSteps × kFeatureDim constants in the Arduino sketch. A global RMS normalization step (target RMS = 0.10, max gain capped at 12×) runs before framing to prevent loud recordings from skewing the feature distribution. Z-score normalization statistics (per-feature mean and std) computed on the training set only are exported with the model so on-device preprocessing matches training exactly.

Model — Three-Layer MLP

The classifier is a compact Multi-Layer Perceptron chosen for minimal FLOP count and simple matrix-multiply operations that map well to embedded arithmetic:

  • Flatten: (101 × 13) → 1,313-element vector.
  • Dense 64 + ReLU — first hidden layer.
  • Dense 64 + ReLU — second hidden layer.
  • Dense 5 (logits) — one score per keyword; argmax at inference.

Trained for 25 epochs with Adam (lr = 1e-3), batch size 64, and sparse categorical cross-entropy loss. Evaluation used per-class precision, recall, F1-score, and a confusion matrix to identify inter-keyword confusions.

Embedded Deployment — C Header Export

After training, the Keras weight matrices were transposed to C row-major layout [output][input] and serialized — together with normalization statistics — into a single header file keyword_linear_model.h. The Arduino sketch (NanoKeywordDemo.ino) includes this header and executes the full forward pass:

  1. Capture 1-second PDM audio frame.
  2. Compute per-frame log-energy features (101 × 13).
  3. Apply z-score normalization using exported kFeatMean / kFeatStd.
  4. Three dense layer multiplications with ReLU gates.
  5. Argmax over logits → keyword label emitted over Serial.

A parallel TFLite path was also produced (keyword_model.tflite + keyword_cnn_energy.h5) to compare the custom header inference route against the standard TFLite Micro runtime.

Key Engineering Decisions

  • Log-energy over MFCC: cuts per-frame compute by ~80% while retaining sufficient spectral envelope information for the narrow five-class vocabulary.
  • MLP over CNN: eliminates the sliding convolution kernel, which is expensive on a device without SIMD audio extensions.
  • C header over TFLite runtime: removes the TFLite Micro interpreter overhead (~8 KB RAM) and produces a single, auditable, dependency-free inference file.
  • Normalization parity: exporting training-set statistics ensures the device sees the same input distribution the model was trained on — critical for avoiding silent accuracy degradation at deployment.

Project 2 of 2 · Motion / IMU

Real-Time Posture Classification via IMU-Driven 1D-CNN

What It Does

A body-posture classifier that reads raw inertial and magnetic field data from the Arduino's onboard IMU and predicts one of five postures — Supine, Prone, Side, Sitting, or Unknown — in real time. Three independent sensor-specific models (accelerometer, gyroscope, magnetometer) are stored on the device simultaneously; the user selects which sensor stream to use for each inference query over serial.

Data Collection

Raw IMU data was logged from the Arduino via a Python base station (base_station.py and logger_stop_on_enter.py) that opened a serial connection at 115200 baud and streamed 10-column CSV rows:

t, ax, ay, az, gx, gy, gz, mx, my, mz

Each CSV file was named to embed its posture label — e.g., supine_trial1.csv, prone_accel.csv — so the dataset builder could infer the ground-truth class from the filename automatically, without a separate annotation file.

Dataset Engineering — Sliding Windows

The dataset builder (build_dataset.py) scanned all raw CSVs and separated each recording into three sensor channels. For each channel, a sliding window approach converted the continuous time-series into fixed-length training samples:

  • Window size: 100 samples (~2 seconds at 50 Hz)
  • Stride: 50 samples (50% overlap for data augmentation)
  • Shape per window: (100 × 3) — 100 time steps × 3 axis values

This produced three independent .npz datasets — dataset_accel.npz, dataset_gyro.npz, dataset_mag.npz — each containing an X array of shape (N, 100, 3) and a label vector y with integer values 1–5. The split was 70 / 15 / 15 (train / val / test) with stratification per posture class.

Normalization — Per-Sensor, Train-Only

Each sensor occupies a completely different physical scale (g-forces vs °/s vs µT), so normalization is computed independently per sensor type. Channel-wise mean and standard deviation are calculated on the training split only, then saved to norm_stats_{sensor}.npz. The same statistics are hard-coded as float arrays in the Arduino sketch:

  • kMean_accel / kStd_accel — for accelerometer windows
  • kMean_gyro / kStd_gyro — for gyroscope windows
  • kMean_mag / kStd_mag — for magnetometer windows

The active pointer (kMean, kStd) is switched at model-load time to match whichever sensor the user selected, ensuring normalization is always paired with the correct model.

Model — Sensor-Specific 1D-CNNs

A lightweight 1D convolutional architecture was chosen over an MLP here because posture patterns manifest as temporal shape in the sensor signal — a pattern CNNs exploit naturally. Each of the three sensor datasets gets its own trained model:

  • Conv1D (16 filters, kernel 5) + ReLU — captures local motion patterns.
  • Conv1D (32 filters, kernel 5) + ReLU — learns higher-order temporal abstractions.
  • GlobalAveragePooling1D — aggregates across the time axis; no fixed-length flattening required.
  • Dense 32 + ReLU — compact intermediate representation.
  • Dense 5 + Softmax — probability over the five posture classes.

All three models were trained with Adam (lr = 1e-3), sparse categorical cross-entropy, up to 100 epochs with early stopping on validation accuracy (patience = 10) to prevent overfitting on the limited dataset. Performance was evaluated with per-class precision, recall, F1, and confusion matrices to identify which postures and which sensors were most discriminative.

TFLite Conversion & C Header Generation

The trained Keras models were converted to TFLite flat-buffer format via a concrete-function converter (convert_to_tflite.py) keeping pure float32 operations — no quantization — to maintain accuracy without the quantization-calibration step. Each .tflite binary was then serialized into a C header as a const unsigned char[] array, producing:

  • posture_model_accel_data.h
  • posture_model_gyro_data.h
  • posture_model_mag_data.h

All three headers are compiled into the same Arduino sketch, and the correct model is loaded into the 32 KB TFLite tensor arena at runtime using modelInit() from the ArduTFLite library.

Arduino Inference Sketch

The sketch (posture_inference.ino) exposes a simple serial menu. On each request:

  1. User sends '1', '2', or '3' to select accelerometer, gyroscope, or magnetometer.
  2. The corresponding TFLite model is loaded into the tensor arena (skipped if already loaded).
  3. 100 IMU samples are collected at ~50 Hz into a float window_buf[100][3].
  4. Each sample is z-score normalized in-place using the sensor's hard-coded mean/std.
  5. modelRunInference() runs the TFLite graph; per-class probabilities are read with modelGetOutput().
  6. The argmax label and confidence are sent back as a PRED,label,prob line that the Python base station parses and prints as a human-readable posture name.

The Python base station (base_station.py) auto-detects the Arduino's serial port, presents the same sensor menu interactively, and translates the numeric label into its string name (Supine, Prone, Side, Sitting, Unknown).

Key Engineering Decisions

  • Three separate models instead of one: each sensor has distinct statistical properties and temporal dynamics. Training sensor-specific models avoids cross-sensor noise and lets each model specialize.
  • 1D-CNN over MLP for motion: posture is encoded in the temporal shape of the signal (e.g., gravity vector direction for accelerometer). Convolutional filters extract that shape directly; a flat MLP would need to learn it implicitly from 300 inputs.
  • TFLite + ArduTFLite over raw C headers: unlike the keyword project, three models must coexist and be hot-swapped at runtime. The TFLite interpreter handles this gracefully; hand-writing three forward-pass C functions would be error-prone and inflexible.
  • 50% window overlap: effectively doubles the training data from the same recorded sessions, improving generalization without requiring additional recording time.
  • Global average pooling: produces a fixed-length representation regardless of window variation, and has zero learnable parameters — reducing model size.

Engineering Takeaways

  • The choice of deployment path (C header vs TFLite runtime) should be driven by how many models need to coexist on the device and whether dynamic model loading is required.
  • Feature engineering for embedded targets is a hardware-aware problem: the right feature is the most compact one that preserves the information the classifier needs, computed within the interrupt budget.
  • Sensor-specific normalization must be computed on training data and exported precisely — even small mismatches between training and inference statistics can silently degrade accuracy without triggering any error.
  • Sliding window overlap is a cheap data augmentation technique for time-series that can meaningfully improve generalization when labeled data is scarce.
