Embedded TinyML Inference Systems

Two sensor modalities — audio and motion — deployed as real-time, on-device neural classifiers on a single Arduino Nano 33 BLE Sense. No cloud. No host compute. Every inference decision made locally in milliseconds.

Overview

This project covers the full TinyML development cycle — data collection, feature engineering, model training, and embedded deployment — applied across two distinct sensing domains on the same constrained hardware. A keyword spotter listens for five spoken commands via the onboard PDM microphone; a posture classifier reads body orientation from the IMU and magnetometer. Together they demonstrate that the same embedded ML pipeline adapts cleanly across fundamentally different sensor types and problem structures.

What I Built

  • A real-time audio keyword spotter that recognizes five spoken commands — all, must, none, never, only — from a continuous microphone stream using a log-energy MLP deployed as a C header file with no runtime dependency.
  • A real-time body posture classifier that identifies five posture states — Supine, Prone, Side, Sitting, Unknown — from 100-sample IMU windows using three sensor-specific 1D-CNNs hot-swapped at runtime via TFLite Micro.
  • End-to-end Python pipelines for both: data collection scripts, feature extraction, normalization, model training with evaluation metrics, and automated export to embedded-ready formats (C header and TFLite binary).

Pipeline

Both systems follow the same four-stage embedded ML pipeline, with each stage adapted to the sensor at hand.

1. Data Collection. Audio clips were recorded as 1-second WAV files at 16 kHz, organized into keyword-named folders for automatic label inference. IMU data was streamed over serial at 115200 baud and logged as 10-column CSVs (t, ax, ay, az, gx, gy, gz, mx, my, mz), with posture labels encoded in the filename.

2. Feature Extraction. For audio, each clip is segmented into 101 frames and converted to a log-energy feature — log(1 + mean|x|) per frame — replicated across 13 dimensions to produce a (101 × 13) tensor. For motion, raw 3-axis sensor values are segmented into 100-sample sliding windows (50% overlap) yielding (100 × 3) tensors, one per sensor type.

3. Training & Evaluation. The audio classifier uses a three-layer MLP (Flatten → Dense 64 → Dense 64 → Dense 5) optimized for minimal FLOP count. The posture classifier uses a 1D-CNN (Conv1D 16 → Conv1D 32 → GlobalAvgPool → Dense 32 → Dense 5) to exploit the temporal shape of inertial signals, with three independently trained models — one per sensor modality. Both were evaluated with per-class precision, recall, and F1-score.

4. Embedded Deployment. The audio model weights are exported as a C header (keyword_linear_model.h) with training-set normalization statistics embedded alongside — no runtime interpreter needed. The three posture models are converted to TFLite flat-buffers and compiled into the Arduino sketch as C arrays; the ArduTFLite library loads the correct model into a 32 KB tensor arena based on the user's sensor selection.

Engineering Focus

  • Hardware-aware feature design. Log-energy was chosen over MFCC for audio because it cuts per-frame compute by ~80% while retaining enough spectral envelope for a five-class vocabulary. Raw IMU windows were used directly for motion because the convolutional filters learn the relevant temporal patterns without a handcrafted transform.
  • Deployment path matched to problem structure. A single audio model with no runtime fits cleanly into a custom C header. Three posture models that need runtime hot-swapping are better served by the TFLite interpreter, which handles model loading and tensor arena management cleanly.
  • Normalization parity. Training-set statistics (mean, std) are exported and hard-coded on the device for both systems. Even small drift between training and inference normalization silently degrades accuracy — exporting them explicitly eliminates that risk.
  • 50% window overlap for motion data. Doubles the number of training samples from the same recording sessions without additional data collection — a practical augmentation for resource-limited embedded datasets.
